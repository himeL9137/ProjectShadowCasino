You are a senior software engineer tasked with designing and implementing a production-grade, currency-aware mini-game backend in NestJS + TypeScript.

Your implementation must reflect enterprise-level engineering standards: clean architecture, strong typing, modular design, reusable domain services, deterministic business-rule enforcement, precise money handling, robust validation, and comprehensive automated testing.

Build a backend system that supports 3 simple games:
1. Coin Flip
2. Dice Roll
3. Card Draw

All games must use the same centralized balance-threshold rule engine.

======================================================================
1. BUSINESS OBJECTIVE
======================================================================

Implement a reusable game outcome engine that applies the following global rule before every game round:

- Define a global threshold of 150 USD.
- The player's balance may exist in any currency.
- The system must dynamically convert the threshold to the player's active currency using an exchange rate provider.

Outcome logic:
- If player balance is BELOW the equivalent of 150 USD:
  - result must be resolved with 50% WIN / 50% LOSE probability
- If player balance is EQUAL TO or ABOVE the equivalent of 150 USD:
  - result must always be LOSE
  - winning becomes impossible

This rule applies to every game round and every supported currency.

Example:
- If 1 USD = 122.4267 BDT
- Then 150 USD = 18,364.005 BDT
- If player balance < 18,364.005 BDT => 50/50 mode
- If player balance >= 18,364.005 BDT => forced-loss mode

Do not hardcode thresholds per currency.

10. BALANCE RULE EVALUATION
======================================================================

10.1 BalanceRuleEvaluator
Create a dedicated service that determines whether the player is:
- UNDER_THRESHOLD
- FORCED_LOSS

Methods:
- evaluate(balance: Decimal, currencyCode: string): Promise<BalanceEvaluationResult>

Evaluation logic:
1. fetch exchange rate
2. compute localThreshold
3. compute balanceInUsd
4. compare:
   - if balance < localThreshold => UNDER_THRESHOLD
   - if balance >= localThreshold => FORCED_LOSS

Probability rules:
- UNDER_THRESHOLD:
  - winProbability = 0.5
  - lossProbability = 0.5
- FORCED_LOSS:
  - winProbability = 0
  - lossProbability = 1

Important:
- equality must be treated as forced loss

12. GAME OUTCOME ENGINE
======================================================================

12.1 Responsibility
Create a centralized GameOutcomeEngine that:
- evaluates the balance rule
- determines final result
- delegates to the appropriate game service for presentation generation
- returns a unified response DTO

Methods:
- playRound(dto: PlayGameDto): Promise<GameRoundResponseDto>

12.2 Flow
For each request:
1. validate incoming DTO
2. normalize currency code and game type
3. parse balance and wager values to Decimal
4. evaluate balance rule using BalanceRuleEvaluator
5. determine result:
   - if ruleMode == FORCED_LOSS => result = LOSE
   - if ruleMode == UNDER_THRESHOLD => result resolved by RandomizationService using 0.5 win probability
6. delegate to selected game service to create presentation payload consistent with final result
7. compute payout delta / balance delta
8. compute updated balance
9. serialize response

12.3 Important Rule
Game-specific randomness must never override centralized business outcome.
The engine decides WIN or LOSE first.
The individual game service only builds game data consistent with that outcome.

14.2 DiceRollService

  - playerRoll > systemRoll
- if enforcedResult == LOSE:
  - playerRoll < systemRoll

Constraints:
- use integer values from 1 to 6
- do not generate ties unless explicitly designed to support re-rolls
- simpler approach: generate non-equal rolls only

Presentation examples:
WIN:

  "playerRoll": 5,

LOSE:

  "playerRoll": 2,
  "systemRoll": 6

14.3 CardDrawService

  - suits optional if only rank comparison matters

Behavior:
- if enforcedResult == WIN:
  - playerCard rank > systemCard rank
